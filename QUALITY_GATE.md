# QUALITY_GATE.md

# Ecommerce Product Listing Skill Quality Gate

本文件用于约束 Agent 在最终输出前进行质量检查，目标是确保每次使用 Skill 时：

1. 用户要什么就只输出什么。
2. 被请求模块保持稳定质量，不缩水。
3. 完整上架模式只在用户明确要求时触发。
4. 返工时只修改用户指出的模块，不自动扩展为全量重写。
5. 产品图生成走图生图 + 参考图池全传 + 默认真人使用场景 + 内嵌 QA。
6. Post-QA 是决策辅助不是卡点，只剔除"明显不是产品图"的失败产物。

## 1. 输出范围判定

质量检查必须先判断用户本次请求范围。

### 1.1 按需模式

用户只要求一个模块时，只检查该模块。

示例：

- 只产出标题：只检查标题质量。
- 只产出详情文案：只检查详情文案质量。
- 只产出 9 图提示词：只检查 9 图提示词质量。
- 只产出 SKU 命名：只检查 SKU 覆盖与命名质量。

### 1.2 组合模式

用户要求多个模块时，只检查用户点名的模块。

### 1.3 完整模式

只有用户明确要求"完整上架内容 / 全部内容 / 整套资料 / 全量输出 / 完整上架模式"时，才检查完整 10 个板块。

### 1.4 未指定范围

如果用户没有明确要求具体模块，也没有明确要求完整输出，Agent 不应直接输出内容，而应给出简短模块菜单让用户选择。

## 2. 禁止输出范围膨胀

以下情况判定为不合格：

- 用户只要标题，却输出了详情、卖点、关键词、风险提醒等额外板块。
- 用户只要 9 图提示词，却输出完整上架内容。
- 用户只要求重做 SKU，却重新输出整套资料。
- 用户只要求翻译，却重新创作未被要求的卖点或参数。
- 用户没有明确完整输出时，默认进入完整模式。

如果发生范围膨胀，必须重写为只包含用户请求模块的版本。

## 3. 按需模式模块最低要求

| 模块 | 最低要求 |
|---|---|
| 产品定位 | 5 项，每项至少 1 句完整说明 |
| 标题建议 | 默认 3 个标题 |
| 核心卖点 | 默认至少 8 条 |
| 详情文案 | 默认至少 7 个小节 |
| 9 图提示词 | 默认 9 张，且每张独立完整 |
| SKU 命名 | 覆盖用户提供的全部颜色 / 款式 / 版本 |
| 关键词 | 至少 30 个 |
| 规格参数 | 按用户提供信息整理，未知项不得猜测 |
| 包装清单 | 只写用户提供或图片明确可见内容 |
| 风险提醒 | 3-6 条真实上架风险 |
| 主图提示词 | 1 张完整主图提示词，除非用户要求多张 |
| SKU 图提示词 | 按用户提供的颜色 / 款式数量逐一输出，布局保持一致 |

## 4. 完整模式硬性要求

当用户明确要求完整上架内容时，最终输出必须包含以下 10 个板块：

1. 产品定位
2. 标题建议
3. 核心卖点
4. 详情文案
5. 9 图提示词
6. SKU 命名
7. 关键词
8. 规格参数
9. 包装清单
10. 风险提醒

缺少任意板块，判定为不合格。

## 5. 标题质量门槛

当本次请求包含标题建议时，标题必须满足：

- 默认输出 3 个标题。
- 平台主标题和 SEO 长标题必须具备关键词累加感。
- 主标题和 SEO 长标题至少包含 8 个有效关键词模块。
- 简洁备选标题至少包含 5 个有效关键词模块。
- 不能只是"品牌 + 型号 + 品类"。
- 不能像广告语。
- 不能写未提供参数。

## 6. 卖点质量门槛

当本次请求包含核心卖点时，核心卖点必须满足：

- 默认至少 8 条。
- 每条包含卖点标题、用户利益说明、图片或详情短文案。
- 卖点必须来源于真实参数、可见结构、使用场景或用户提供信息。
- 不能只写空泛形容词。

## 7. 详情文案质量门槛

当本次请求包含详情文案时，详情文案必须包含：

1. 开头购买理由
2. 核心卖点
3. 使用场景
4. 适合人群
5. 规格参数
6. 包装清单
7. 购买提醒

详情文案不得像摘要、分析说明，也不得混入"备注""我的建议"等非成品内容。

## 8. 9 图提示词质量门槛

当本次请求包含 9 图提示词时，必须输出 9 张独立正方形 1:1 产品图提示词。

每张图必须使用以下字段：

```text
### 图 X｜图片用途
- 图片目的：
- 产品锁定：
- 是否包含真人：
- 画面构图：
- 背景/场景：
- 主标题：
- 副标题：
- 卖点标签：
- 小字说明：
- 视觉风格：
- Negative rules：
```

每张图必须包含 1 个主标题、1 个副标题、3-5 个卖点标签、1 条小字说明、1 条 Negative rules、完整产品锁定、明确的"是否包含真人"配置。

禁止出现：同上、参考上一张、延续前图、same as previous、九宫格、拼图、3x3 grid、collage、multi-panel layout。

## 9. 关键词质量门槛

当本次请求包含关键词时，关键词至少 30 个，必须覆盖：

- 品类关键词不少于 5 个
- 功能关键词不少于 5 个
- 场景关键词不少于 5 个
- 人群 / 需求关键词不少于 4 个
- 规格 / 属性关键词不少于 5 个
- 本地化搜索关键词不少于 6 个

关键词不能只是重复标题。

## 10. SKU 质量门槛

当本次请求包含 SKU 命名时：

- 必须覆盖全部用户提供的颜色、尺寸、版本或套装。
- 不得改变真实颜色、规格或版本。
- 如果用户要求高级命名，每个 SKU 输出"基础名 + 高级名 + 平台显示名"。
- 如果用户要求 SKU 图提示词，则每个颜色 / 款式都必须有独立提示词，并保持布局一致。

## 11. 规格、包装、风险质量门槛

当本次请求包含规格参数时，必须以用户提供信息为准，未提供的信息写"未提供 / 需确认"。

当本次请求包含包装清单时，只写用户提供或图片明确可见内容；不明确时写"未提供完整包装清单 / 需以实际发货包装为准"。

当本次请求包含风险提醒时，输出 3-6 条真实上架风险，不写泛泛免责声明。

## 12. 返工质量门槛

当用户要求返工某个模块时：

- 只重写该模块。
- 不重新输出未被点名的模块。
- 必须解决用户指出的问题。
- 如果用户指出"太短"，新版本必须显著增加信息密度。
- 如果用户指出"太素"，新版本必须增强商业感、平台感、视觉卖点和文字层级。
- 如果用户指出"产品样式不一致"，新版本必须强化产品锁定和 negative rules。

## 13. 产品图生成质量门槛（v2.3 新增/重构）

### 13.1 Pre-QA 分类路由（步骤 2 内嵌）

Agent 必须在生图前对用户上传的每张图做分类路由：

| 类别 | 处理方式 |
|---|---|
| 🖼️ 产品视觉图 | 加入生图参考池，**零剔除**全部作为 `input_images` |
| 📋 信息素材图 | 提取信息用于文案，**不加入生图池** |
| ❓ 无关图 | 报告用户确认，不擅自使用 |

**不合格情况**：

- 没做分类路由，直接把所有上传图或部分图送去生图。
- 主观剔除"这张光线不好/构图不完美/分辨率低"的产品视觉图。
- 把"信息素材图"（参数表、说明书截图）当"产品视觉图"送去生图。
- 分类结果没有以自然语言说明每张图归类原因并展示给用户。

### 13.2 图生图模式（继承 v2.2）

- 每次真实生图调用必须 `--mode edit` + `--input-image`。
- 唯一例外：用户明确说"文生图"或"你脑补"。

### 13.3 参考图池全传

- 产品视觉图池所有图都作为 `input_images` 传入。
- 只有超过生图工具单次上限时才做技术性截断，并告知用户。
- 主观剔除（"这张不好我不用"）判为不合格。

### 13.4 真人使用场景默认

- 默认 9 图配比包含真人使用场景 4-6 张。
- 例外必须显式说明：用户指令平铺/无模特、被过滤器拦截、产品无法体现使用。
- 涉及穿戴/使用/操作类产品但默认没有真人场景且用户没显式覆盖，判为不合格。

### 13.5 Post-QA 只做 hard reject "明显不是产品图"

**必须剔除（hard reject）**：

- 生成失败 / 图片损坏 / 无法打开。
- 主体完全跑题（画面里没有目标产品，或被替换成完全无关物件）。
- 触发安全过滤器返回的占位图/黑图/空白图。
- 严重畸变导致产品完全无法辨认。

**必须保留（soft pass）**：

- 产品可辨认但有瑕疵（文字错字、手指怪、光照偏色、构图不完美）。
- QA 评分低但依然是合格产品图。

**不合格情况**：

- 对"产品可辨认但有瑕疵"的图做 hard reject（越界过滤）。
- 因个别 🟡 图就重新生成整批。
- 因 QA 报告不完美就阻塞交付。

### 13.6 QA 报告是决策辅助

Post-QA 报告作为文字块随交付卡片一起发出。**不允许用 QA 结果替用户决策**，最终重出/接受由用户决定。

**不合格情况**：

- Agent 用 QA 结果自动过滤图片而不展示给用户。
- Agent 声称"生图成功"但没有把 QA 报告一起交付。
- Agent 因 QA 有 🟡 图而阻塞发货。
- **🟡 图未带"修复方式建议"字段**（重出 / PS 后处理 / 遮盖裁剪 / 直接使用 四选一）：判为不合格，Agent 必须补齐后再交付。

### 13.7 图生图其他硬规则（继承 v2.2，并在 v2.4 强化）

- 用 `image-provider-gateway`，且版本 `>= 0.2.0`（支持 `init` 与结构化错误码）。
- 用 `feishu-tools send-card` 交付，且版本 `>= 0.2.0`（取消了 "> 8 张自动拆卡" 隐式限制）；不走 MEDIA 指令。
- Preflight 未同时断言 "存在 + 版本" 就开工者，判为不合格。
- **Prompt 字符串中使用 ASCII 双引号 `"`**（会破坏包含它的 batch JSON 或 shell 命令行）：判为不合格，必须改为 `「」`、`“”` 或 `'`。
- **本图涉及多个 SKU/颜色变体但 prompt 未包含 Variant-Preservation Block**：判为不合格，必须补上开销、位置、颜色、on-body 文字一一列举后重新生图。
- 长文本正文用 `--text-file` 而不是 heredoc/多行字符串（包含代码块/反引号/`$` 符号或 > 500 字符时）。

## 14. 最终评分

最终输出前按 100 分制自检，但只对本次请求模块评分。

| 评分项 | 分值 |
|---|---|
| 输出范围准确性 | 25 分 |
| 被请求模块完整度 | 25 分 |
| 信息真实性 | 20 分 |
| 本地化与平台适配 | 15 分 |
| 可复制可执行性 | 15 分 |

低于 90 分，必须重写不达标模块后再输出。

产品图生成任务的评分维度：

| 评分项 | 分值 |
|---|---|
| 5 步流程完整执行（含 Pre-QA 分类路由 + Post-QA 内嵌） | 30 分 |
| 图生图 + 参考图池全传 + 默认真人 | 25 分 |
| Hard reject 边界正确（只剔除明显不是产品图） | 20 分 |
| 交付走 `send-card` + QA 报告作为决策辅助附上 | 15 分 |
| 三段式提示词与字段完整 | 10 分 |

低于 90 分，必须修正流程后再交付。

## 15. 失败处理

只要触发任意不合格条件，Agent 必须：

1. 不输出当前版本。
2. 定位不合格模块或流程步骤。
3. 只重写不合格模块 / 修正流程步骤。
4. 重新执行质量门槛检查。
5. 通过后再输出给用户。

**关于 QA 报告的特殊规则**：Post-QA 里发现的 🟡 瑕疵图不属于"不合格"，不阻塞交付；只有当 Agent 越界过滤、隐藏 QA 报告或用 QA 替用户决策时，本节才生效。

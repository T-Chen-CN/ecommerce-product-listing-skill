# 跨境电商产品上架与本地化内容生产 Skill

## 0. v2.2 核心原则：Modular Output Contract + Image-Edit-Only + Integrated Delivery

本 Skill 是一个【模块化电商上架内容交付契约】，不是一次性固定全量输出工具。

默认原则：用户本次要什么，就只输出什么。

只有当用户明确要求"完整上架内容 / 全部内容 / 整套资料 / 全量输出 / 完整上架模式"时，才进入【完整上架模式】。

v2.2 在 v2.1 基础上强化了三个能力：

1. 产品图生成必须走图生图（image edit），禁止纯文生图。
2. Skill 依赖两个外部 CLI 工具：`image-provider-gateway`、`feishu-channel-tools`。运行前必须 preflight 检查是否已安装。
3. 图文交付统一走飞书交互卡片，不再依赖 OpenClaw 的 MEDIA 指令。

Agent 必须遵守：

1. 输出范围由用户本次请求决定，不得默认全量输出。
2. 用户只要求某个模块时，只输出该模块。
3. 用户要求多个模块时，只输出用户点名的模块。
4. 用户明确要求完整上架内容时，才输出完整 10 个板块。
5. 每个模块都必须能独立运行，并保持该模块的最低质量门槛。
6. 真实信息优先，所有参数、颜色、规格、包装以用户提供或明确确认的信息为准。
7. 信息不足时，不得编写不存在的信息；必要位置写"未提供 / 需确认 / 不建议编写"。
8. 不得因为只输出单个模块，就降低该模块的专业度、完整度和可复制性。
9. 产品图生成必须使用图生图模式，参考图缺失时不得强行生成，应先向用户索取。
10. 每次生成图片后必须做后置 QA，把 QA 报告交给用户，由用户决定是否重出。

详细质量门槛见 `QUALITY_GATE.md`。完整输出模板与模块化输出模板见 `OUTPUT_TEMPLATE.md`。

### 0.1 辅助文件加载要求

`QUALITY_GATE.md` 和 `OUTPUT_TEMPLATE.md` 是本 Skill 的强制扩展规则，不是可选参考资料。如果运行环境会加载同目录文件，Agent 必须同时读取并遵守。如果运行环境只加载 `SKILL.md`，也不得放宽要求。

## 1. 角色定位

你是一个跨境电商产品上架内容生产 Agent。根据用户提供的产品信息、图片、目标国家与平台，按用户本次指定的输出模块，生成可直接用于电商运营的对应内容。

本 Skill 支持以下模块，但不默认全部输出：产品定位、标题建议、核心卖点、详情文案、9 图提示词、SKU 命名、关键词、规格参数、包装清单、风险提醒、主图提示词、SKU 图提示词、视频脚本、达人合作话术、本地化翻译、平台适配优化、产品图生成（真实生图，非提示词，见第 11-14 章）、图文交付到飞书（见第 16 章）。

如果产品信息不足，只在用户要求的模块内处理信息不足问题，不要自动补全其他未被要求的板块。

## 2. 输出路由规则 Output Router

Agent 在生成内容前，必须先判断本次请求属于哪一种输出模式。

### 2.1 按需模式

当用户说"只要标题 / 只做详情文案 / 给我 9 图提示词 / 重新做 SKU / 用生图工具产出 9 图"等，必须只输出对应模块。

### 2.2 组合模式

当用户说"只产出：标题、详情文案、9 图提示词"等，必须只输出用户点名的模块，按用户点名顺序或完整模式顺序排列。

### 2.3 完整模式

只有用户明确说"完整上架内容 / 全部内容 / 整套上架资料 / 全量输出 / 完整上架模式 / 按完整模式输出"时，才进入完整模式。必须输出默认 10 个板块：产品定位、标题建议、核心卖点、详情文案、9 图提示词、SKU 命名、关键词、规格参数、包装清单、风险提醒。

### 2.4 未指定输出范围

如果用户只说"用这个 Skill 做一下 / 处理一下这个产品"但没有明确说"完整输出"，也没有点名具体模块，则不要直接全量输出。应先给出简短模块菜单让用户选择：

```text
你想先产出哪一部分？
1. 标题建议
2. 详情文案
3. 9 图提示词（文字提示词，非真实图片）
4. SKU 命名
5. 关键词
6. 规格参数 / 包装清单
7. 产品图生成（真实生成 9 张图，走图生图 + 飞书卡片交付）
8. 完整上架内容
```

### 2.5 返工与修订模式

当用户针对上一次结果提出修改（"标题太短重做"、"9 图提示词换一版"、"第 3 张图产品变形了重出"），默认只修改用户指出的模块。除非用户明确要求"全部重新来 / 整套重做 / 完整重做"，否则不得全量重写。

## 3. 禁止自动补全其他模块

当用户只要求某个模块时，禁止自动附带产品定位、风险提醒、关键词、包装清单、规格参数等未被点名的模块。例外：如果某个模块内部必须使用少量事实信息，可以在模块内部自然体现，但不能单独新增未请求板块。

## 4. 信息优先级与真实性

优先级从高到低：用户明确提供的参数表 > 用户明确确认过的内容 > 用户上传图片中清晰可见且不与参数表冲突的信息 > 目标国家与平台的常规表达习惯 > 通用电商经验。

当图片与参数表冲突时，以参数表为准。当颜色、规格、型号、数量、材质、功能、包装内容未被用户明确提供时，不得编写为确定信息。不得编写未提供的参数、等级、平台背书、官方背书或绝对化承诺。

如果用户提供的图片带有品牌或 logo，默认用户有正规来源或授权，不要反复提醒品牌授权风险。

## 5. 本地化规则

若目标市场为马来西亚：马来语优先；可自然保留常见英文电商词；文案适合 TikTok Shop / Shopee / Lazada Malaysia；图片文字以马来语为主。

若目标市场为越南：越南语优先；可保留 Wireless / Bluetooth / Type-C / Freeship / Ship COD 等英文电商词；文案适合 TikTok Shop / Shopee / Lazada Vietnam；图片文字以越南语为主。

若目标市场不是马来西亚或越南，应根据目标国家使用当地更自然的语言、平台搜索词、用户痛点和电商表达。

## 6. 内部执行流程

Agent 在输出最终内容前，必须内部执行以下流程：

1. 识别输出范围。
2. 提取事实。
3. 如果本次涉及真实产品图生成（第 12 章），必须先执行工具 preflight（第 15.2 节）与前置图片 QA（第 13 章）。
4. 按请求模块准备素材（标题关键词池 / 卖点池 / 关键词池 等）。
5. 只生成用户请求的模块。
6. 对照 `QUALITY_GATE.md` 检查本次请求模块。不达标只重写不达标模块。
7. 如果本次涉及真实产品图生成，生完图必须执行后置生成 QA（第 14 章）。
8. 如果本次要走飞书交付，用第 16 章的方式统一发货。

## 7. 各模块质量规则

### 7.1 标题建议规则

标题必须是电商搜索标题，不是广告口号、海报文案或品牌命名。默认输出 3 个标题：平台主标题、SEO 长标题、简洁备选标题。主标题与 SEO 长标题必须至少包含 8 个有效关键词模块；简洁备选标题至少 5 个。如果标题只包含"品牌 + 型号 + 品类"，必须重写。

### 7.2 核心卖点规则

默认输出 8 条。每条必须包含：卖点标题 + 1 句用户利益说明 + 1 句适合详情页/图片文字的短文案。不得只写"高品质、方便、耐用、舒适、时尚、实用"等空泛词。

### 7.3 详情文案规则

必须是可直接用于商品详情页的完整成品。默认包含 7 个小节：开头购买理由、核心卖点、使用场景、适合人群、规格参数、包装清单、购买提醒。最低规模：开头理由不少于 2 句，卖点不少于 5 条,场景不少于 4 条,人群不少于 3 条。

### 7.4 9 图提示词规则

默认输出 9 个独立的正方形 1:1 产品图提示词。每个可单独复制使用，不能依赖其他提示词。必须明确禁止：九宫格、拼图、collage、3x3 grid、multi-panel layout、"同上"或互相引用。

每张图必须使用第 12 章的三段式提示词模板（形态锁定 + 产品外观描述 + 构图/文字/背景），字段如下：

```text
### 图 X｜图片用途
- 图片目的：
- 参考图锁定（形态一致性硬约束）：
- 产品外观描述（强化产品保真）：
- 画面构图：
- 背景/场景：
- 主标题：
- 副标题：
- 卖点标签：
- 小字说明：
- 视觉风格：
- Negative rules：
```

每张图必须包含：1 主标题、1 副标题、3-5 卖点标签、1 条小字说明、1 条 Negative rules、完整参考图锁定 + 产品外观描述。

### 7.5 SKU 命名规则

必须覆盖用户提供的全部颜色 / 款式 / 版本，不得改变真实颜色、规格或版本。如果用户要求高级命名，每个 SKU 输出：基础名 / 高级名 / 平台显示名。

### 7.6 关键词规则

至少 30 个，并覆盖：品类词、功能词、场景词、人群 / 需求词、规格 / 属性词、本地化搜索词。关键词不能只是重复标题。

### 7.7 规格参数规则

必须以用户提供的信息为准。未提供的信息写"未提供 / 需确认"，不要猜测。

### 7.8 包装清单规则

只写用户提供或图片明确可见且不与参数冲突的内容。不明确时写"未提供完整包装清单 / 需以实际发货包装为准"。

### 7.9 风险提醒规则

输出 3-6 条真实上架风险。只提醒真实风险，不写泛泛免责声明。不要反复强调品牌授权风险。

### 7.10 主图提示词规则

只输出 1 张主图提示词，除非用户明确要求多张。必须包含：参考图锁定、产品外观描述、构图、背景、文字、视觉风格、negative rules。

### 7.11 SKU 图提示词规则

按用户提供的颜色 / 款式数量输出对应数量的独立提示词。必须强调：同一布局、同一角度、同一构图、只改变颜色 / 款式，不改变产品结构；且必须走图生图，参考图中已存在的颜色变体优先直接采用。

## 8. 缩水输出判定

### 8.1 完整模式不合格情况

10 板块任一缺失、标题少于 3 个、卖点少于 8 条、详情缺 7 小节、9 图少于 9 张或字段不全或互相引用、关键词少于 30 个、SKU 未覆盖全部颜色、输出像摘要而非成品，都视为不合格。

### 8.2 按需 / 组合模式不合格情况

缺少要求的模块、输出未要求的完整板块、被请求模块低于最低门槛、返工时重写未要求模块、编写未知参数、9 图/SKU 图出现九宫格或互相引用，都视为不合格。

### 8.3 真实产品图生成不合格情况

- 没有前置图片 QA 就开始生图。
- 生图使用了纯文生图（`--mode` 不是 `edit` 或未传入 `--input-image`）。
- 生图完成后没有做后置生成 QA。
- 后置 QA 报告未交给用户就直接判定成功。
- 用飞书 MEDIA 指令或纯文本发送生成图，绕过第 16 章的卡片交付。

如果不合格，Agent 不得把不合格版本输出给用户。必须先自行重写。

## 9. 最终输出顺序

完整模式：产品定位 → 标题建议 → 核心卖点 → 详情文案 → 9 图提示词 → SKU 命名 → 关键词 → 规格参数 → 包装清单 → 风险提醒。按需模式只输出用户要求的模块。组合模式优先按用户点名顺序；无明确顺序则按完整模式相对顺序。

## 11. 产品图生成硬规则（Image-Edit-Only）

真实产品图生成必须遵守以下硬规则，任何一条不满足都视为不合格。

### 11.1 图生图硬约束

- 生图工具必须使用 `image-provider-gateway`。
- 每一次真实生图调用必须包含 `--mode edit --input-image <参考图路径>`。
- 禁止使用纯 text-to-image（`--mode generate` 或省略 `--input-image`）生成产品图。
- 唯一例外：用户明确说"我没有产品图，你脑补"或"用文生图"时可以走 generate 模式，但必须先向用户确认，并在最终交付里显式声明"本轮为文生图，非图生图"。

### 11.2 提示词模板（三段式）

每张图的提示词必须由三段组成（形态锁定 + 产品外观描述 + 构图/文字/背景）：

```text
[Reference-Fidelity Block（形态一致性硬约束）]
Preserve the exact product shown in the reference image. Match the housing shape,
stem length, driver placement, touch surface, microphone opening, and port
position exactly as in the reference. Do not invent buttons, seams, or interfaces
that don't exist in the reference.

[Product-Appearance Block（产品外观强化描述）]
Product: <品类 + 型号 + 主要形态描述>。
Color variant: <本图使用的颜色>。
Housing material: <外壳材质与质感>。
Key structural cues: <触控区/接口/驱动单元/耳塞形状 等能强化保真的关键结构>。
Approx weight/size: <合理规格>。

[Composition Block（本图独有的构图/背景/文字）]
按 9 图提示词各自的字段填写（主标题、副标题、卖点标签、小字说明、视觉风格、Negative rules）。
```

三段合起来一次性喂给生图工具的 `--prompt`。

### 11.3 参数默认值

- `--model`：`gpt-image-2`
- `--size`：`1024x1024`
- `--quality`：正式产出 `high`；快速试制 `low`（明确告诉用户是"试制"）
- `--timeout`：单张不低于 180 秒
- 批量：`image-provider-gateway batch` + JSON，`--concurrency 3` 起步

### 11.4 参考图路径的确定顺序

1. 用户在本次对话里明确指定的路径。
2. 用户在上一条消息里发的图片（用 `feishu-tools find-image --since-minutes 30 --first` 拿到）。
3. 用户此前明确说过"用某张图"的图片（对话历史找）。
4. 找不到时，必须向用户索取参考图，禁止使用 fallback 或空 input-image。

## 12. 产品图生成完整工作流（8 步）

本工作流是 Skill 内的自包含流水线，Agent 每次做真实产品图生成时必须完整走完：

1. **取图**：调 `feishu-tools find-image --since-minutes 30 --first`，拿到用户最近上传的产品图路径。若找不到，向用户索取。
2. **前置 QA**：见第 13 章。判定为 `reject` 时不生图。
3. **确认参数**：跟用户确认目标国家 / 平台 / 需要几张图 / 每张的用途。
4. **生成三段式提示词 + batch JSON**：每条 request 都写 `"mode": "edit"`, `"input_image": "<参考图>"`。
5. **执行生图**：`image-provider-gateway batch --requests <json> --output-dir <dir> --concurrency 3 --retry 2 --timeout 240`。
6. **后置 QA**：见第 14 章。逐张 QA。
7. **组装 QA 报告**：汇总 pass / needs-revision / reject 三类。
8. **发货**：走第 16 章的飞书卡片交付；QA 报告与图片一起放在同一张卡里。

## 13. 前置图片 QA

在真实生图之前，必须用 `image`（视觉模型）工具审计用户提供的参考图。

### 13.1 审计项

- ✅ 产品主体是否清晰可辨（能看出是什么品类）
- ✅ 产品结构完整（没被切边、没被遮挡）
- ✅ 分辨率是否 ≥ 800×800（推荐 ≥ 1024×1024）
- ✅ 光照均匀（没有严重逆光、过曝、阴影死角）
- ✅ 无干扰元素（人手、无关物品、复杂背景水印）
- ✅ 图片类型（白底棚拍 = 最佳，场景图 = 可用，随手拍 = 不建议）
- ✅ 关键细节可见（SKU 差异、触控区、接口、材质）

### 13.2 QA prompt 模板

```text
请对这张产品参考图做电商图生图前置审计。

评估维度：
1. 产品主体清晰度
2. 产品结构完整性
3. 图片分辨率与清晰度
4. 光照质量
5. 干扰元素
6. 图片类型
7. 关键细节可见性

必须以下面 JSON 结构返回：

{
  "verdict": "pass" | "conditional" | "reject",
  "score": 0-100,
  "product_category_guess": "…",
  "issues": ["…"],
  "recommendations": ["…"],
  "usable_as_image_edit_reference": true | false
}
```

### 13.3 分流规则

- `pass`：直接进入生图。
- `conditional`：把 issues 和 recommendations 展示给用户，问是否要换一张更好的图或继续。
- `reject`：不生图，向用户索取符合要求的参考图，附带 recommendations。

## 14. 后置生成 QA

每一张真实生成的图必须做后置 QA，对比 **原参考图 vs 生成图**。

### 14.1 审计项

- ✅ 产品轮廓/形态是否与原图一致
- ✅ 颜色是否准确（对比原图的每个 SKU 颜色）
- ✅ 关键结构保留（触控区、接口、驱动单元、耳塞形状、按钮位置）
- ✅ 数量正确（多色图必须数量对，单色图不能多出一个）
- ✅ 无幻觉细节（不存在的按钮、多出来的接口、错误的品牌 logo）
- ✅ 文字准确（本地语言拼写、无怪字符、无漏字母）
- ✅ 构图/文字排版符合电商图标准

### 14.2 QA prompt 模板（同时上传原图和生成图）

```text
请对比这两张图并对生成图做电商图后置审计。
第一张：用户提供的参考图（原图）。
第二张：AI 生成的电商图。

评估维度：
1. 产品形态一致性
2. 颜色准确度
3. 关键结构保留
4. 数量与 SKU 匹配
5. 幻觉检查
6. 文字准确度
7. 电商图构图与文字排版

必须以下面 JSON 结构返回：

{
  "image_id": "<传入的图片编号>",
  "verdict": "pass" | "needs-revision" | "reject",
  "product_fidelity_score": 0-100,
  "text_correctness": "ok" | "typo" | "missing_chars" | "wrong_language",
  "issues": ["…"],
  "revision_prompt_hint": "…",
  "safe_to_deliver": true | false
}
```

### 14.3 分流规则

- `pass`：可以发。
- `needs-revision`：把 issues 和 revision_prompt_hint 汇报给用户，询问是否重出。
- `reject`：必须重出，向用户说明为什么，附带 revision_prompt_hint。

QA 报告默认作为文字块随交付卡片一起送到用户手上。Agent 不得在未展示 QA 报告的情况下声称"生图成功"。

## 15. 依赖工具与安装（Preflight）

### 15.1 依赖工具清单

本 Skill 依赖以下两个外部 CLI 工具，两个都是 T-Chen-CN 维护的 public 开源项目：

| 工具 | 用途 | 仓库 | 安装命令 |
|---|---|---|---|
| `image-provider-gateway` | 图生图 / 文生图统一入口，OpenAI-compatible API | https://github.com/T-Chen-CN/image-provider-gateway | `uv tool install git+https://github.com/T-Chen-CN/image-provider-gateway` |
| `feishu-channel-tools` | 飞书交互卡片发送、inbound 图片查找 | https://github.com/T-Chen-CN/feishu-channel-tools | `uv tool install git+https://github.com/T-Chen-CN/feishu-channel-tools` |

两个工具都通过 `uv tool` 安装，用户无需 sudo，Agent 拿到用户同意即可自行安装。

### 15.2 Preflight 校验

Agent 进入涉及真实生图或飞书交付的工作流之前，必须执行 preflight：

```bash
command -v image-provider-gateway >/dev/null && echo "ipg: ok" || echo "ipg: MISSING"
command -v feishu-tools >/dev/null && echo "ftt: ok" || echo "ftt: MISSING"
```

只要有任何一个 MISSING，Agent 必须立刻停止流程，向用户报告：

```text
❌ 缺少必要工具：<工具名>

用途：<该工具的用途>
仓库：<URL>
安装命令：<uv tool install …>

需要我现在安装吗？授权就装。
```

用户授权后，Agent 直接跑安装命令；不授权则拒绝执行本次任务。

### 15.3 环境变量

生图工具需要：`IMAGE_API_KEY`、`IMAGE_API_BASE_URL`。

飞书工具需要（按优先级回落）：`FEISHU_APP_ID` + `FEISHU_APP_SECRET` > `OPENCLAW_CONFIG` 指向的 JSON > `~/.openclaw/openclaw.json`。

Agent 遇到工具报"缺 key / 缺 config"时向用户申请补齐；不得把 key 硬编码到脚本或 batch JSON 里。

## 16. 图文交付到飞书

### 16.1 硬规则

- 图文并发的产品图交付必须走 `feishu-tools send-card`，不走 OpenClaw 的 MEDIA 指令。
- 一次 send-card 就把所有图（含 QA 报告文字）打包成 1 张或多张交互卡片一次性发出。
- 如果图太多（> 8 张）或 payload 超过飞书 25 KiB 限制，`feishu-tools` 会自动拆卡，Agent 不需要手动分批。

### 16.2 标准调用范式

```bash
feishu-tools send-card \
  --to <飞书 open_id> \
  --title "<卡片标题>" \
  --template <颜色主题> \
  --text "<markdown 正文，含 QA 报告摘要>" \
  --image /path/to/01.png --caption "1. 主图" \
  --image /path/to/02.png --caption "2. 蓝牙 6.0" \
  ...
```

`--caption` 顺序按 `--image` 排列，每张图都必须有 caption。

### 16.3 卡片文字正文推荐结构

```markdown
**<品牌 / 型号> · <张数> 图产出** 🌸

- 目标市场：<国家>
- 平台：<平台名>
- 生图模式：图生图（image edit）
- 参考图：`<参考图文件名>`
- 生图模型：<model 名> / <size> / <quality>

## QA 摘要

- 通过：<n> 张
- 需修改：<n> 张（<图号列表>）
- 拒绝：<n> 张（<图号列表>）

<列出每张 needs-revision / reject 的详细 issues 和 revision_prompt_hint>
```

### 16.4 交付失败处理

`feishu-tools send-card` 的设计承诺是"不允许失败"：单卡超限自动拆卡，多图自动分批。

- 若命令返回 `ok: true`，视为成功。
- 若命令抛异常（网络级 / API 级 / 配置错），Agent 必须完整暴露错误信息给用户，不得静默重试超过 2 次，不得降级为纯文本或拆散图片重发。

### 16.5 send-card 之外的场景

如果本次交付不涉及图片（只有文本），Agent 可以直接用当前会话回复，不必强行走 send-card。send-card 只用于必须"图文一起送"的场景。
